<div className="rs-header">
  <h1 className="rs-title">Design for the Content, Not the Device</h1>
  <p className="rs-description">
    There is no fixed set of screens to design for anymore, and there never really was. A layout that only works at a handful of famous device widths is a layout that breaks the moment someone opens a split-view window, rotates a foldable, or reads at 200 percent zoom. This system takes the opposite stance: start from the smallest reasonable screen, let the content decide where it needs to change, and reach for a fixed breakpoint last, not first.
  </p>
</div>

<div className="rs-section">
  <h2 className="rs-section-title"><span className="material-symbols-outlined">smartphone</span>Mobile First, Content First</h2>
  <p className="rs-lede">
    Mobile first is not a phone rule, it is a discipline. Designing the small screen first forces the hardest question early: what is genuinely essential here. Everything that survives that constraint is the core of the experience, and everything a wider screen adds is a progressive enhancement layered on top of a base that already works. Starting wide and squeezing down does the reverse: it treats the small screen as an afterthought, and the afterthought is where most people actually are.
  </p>

  <div className="rs-editorial">
    <div className="rs-editorial-card">
      <div className="rs-editorial-icon"><span className="material-symbols-outlined">vertical_align_bottom</span></div>
      <div className="rs-editorial-label">Base is the floor</div>
      <p className="rs-editorial-text">Write the small-screen layout with no media query at all. That is the default every browser gets, and it must stand on its own before a single breakpoint is added.</p>
    </div>
    <div className="rs-editorial-card">
      <div className="rs-editorial-icon"><span className="material-symbols-outlined">trending_up</span></div>
      <div className="rs-editorial-label">Enhance upward</div>
      <p className="rs-editorial-text">Each min-width query only adds. More columns, more density, a sidebar that was stacked. Never subtract from wide to reach narrow. Enhancement, not repair.</p>
    </div>
    <div className="rs-editorial-card now">

    </div>

        <td><code>--breakpoint-lg</code></td>
        <td>992px</td>
        <td><span className="rs-role">Laptops and up.</span> Persistent sidebars, three-column grids, the full desktop chrome.</td>
      </tr>
      <tr>
        <td><code>--breakpoint-xl</code></td>
        <td>1200px</td>
        <td><span className="rs-role">Wide desktops.</span> Content is comfortable, not stretched. A good place to cap line length and center the shell.</td>
      </tr>
      <tr>
        <td><code>--breakpoint-2xl</code></td>
        <td>1366px</td>
        <td><span className="rs-role">Very wide displays.</span> The top of the scale. Add breathing room and max-widths, not more columns for their own sake.</td>
      </tr>
    </tbody>
  </table>

  <div className="rs-note">
    <div className="rs-note-title"><span className="material-symbols-outlined">bolt</span>These are the only breakpoints</div>
    <div className="rs-note-body">
      Five tokens, published as CSS custom properties: <code>--breakpoint-sm</code> through <code>--breakpoint-2xl</code>. There is no <code>xs</code> token, because the base layout below <code>sm</code> needs no query to name it. Do not invent a sixth threshold for one stubborn component; solve that component with fluid sizing or a container query instead. Note that a media condition cannot read a custom property, so a query uses the literal pixel value, for example <code>@media (min-width: 769px)</code>, while the token remains the single source of truth those numbers come from.
    </div>
  </div>
</div>

<div className="rs-section">
  <h2 className="rs-section-title"><span className="material-symbols-outlined">devices</span>Media Queries or Container Queries</h2>
  <p className="rs-lede">
    For a decade a media query was the only tool, so it did every job: it asked how wide the whole viewport was and changed everything based on the answer. That works for the page shell, but it is the wrong question for a component. A card does not care how wide the window is; it cares how much room it was given. A container query lets an element respond to the size of its own container instead of the viewport, so the same card lays out one way in a narrow sidebar and another way in a wide main column, with no knowledge of the page around it. This is the shift from page-level to component-level responsiveness, and it is the modern default for anything reusable. Container queries are Baseline as of 2023 and safe to use in current browsers.
  </p>

  <div className="rs-compare">
    <div className="rs-compare-col">
      <div className="rs-compare-tag"><span className="material-symbols-outlined">fullscreen</span>Viewport level</div>
      <div className="rs-compare-name">Media query</div>
      <div className="rs-compare-q">@media (min-width: 992px)</div>
      <ul className="rs-compare-list">
        <li>Asks about the whole viewport.</li>
        <li>Right for the page shell: global grid, top navigation, where the sidebar lives.</li>
        <li>Also the home of preference and capability queries: <code style={{ fontFamily: "'SF Mono', monospace" }}>prefers-color-scheme</code>, <code style={{ fontFamily: "'SF Mono', monospace" }}>prefers-reduced-motion</code>, <code style={{ fontFamily: "'SF Mono', monospace" }}>pointer</code>, <code style={{ fontFamily: "'SF Mono', monospace" }}>hover</code>.</li>
        <li>Keyed to the five breakpoint tokens.</li>
      </ul>
    </div>
    <div className="rs-compare-col modern">
      <div className="rs-compare-tag"><span className="material-symbols-outlined">responsive_layout</span>Component level</div>
      <div className="rs-compare-name">Container query</div>
      <div className="rs-compare-q">@container (min-width: 24rem)</div>
      <ul className="rs-compare-list">
        <li>Asks how much space this element was actually given.</li>
        <li>Right for reusable components: cards, tiles, media objects, widgets that appear in many slots.</li>
        <li>The parent opts in with <code style={{ fontFamily: "'SF Mono', monospace" }}>container-type</code>; children can size to <code style={{ fontFamily: "'SF Mono', monospace" }}>cqw</code> units.</li>
        <li>The component becomes portable: drop it anywhere and it adapts to the slot.</li>
      </ul>
    </div>
  </div>

  <div className="rs-note">
    <div className="rs-note-title"><span className="material-symbols-outlined">lightbulb</span>The dividing line</div>
    <div className="rs-note-body">
      Ask what the thing responds to. If it is the page as a whole, use a media query against a breakpoint token. If it is a component that must work in more than one slot, give its wrapper a container and let it respond to its own width. A capability or preference, dark mode, reduced motion, coarse pointer, is always a media query, never a container query.
    </div>
  </div>
</div>

<div className="rs-section">
  <h2 className="rs-section-title"><span className="material-symbols-outlined">waves</span>Fluid and Intrinsic Layouts</h2>
  <p className="rs-lede">
    The best breakpoint is the one you never had to write. Between the structural thresholds, layout should flow, not jump. Intrinsic web design, Jen Simmons' name for this way of working, leans on what CSS can already do on its own: a grid that fits as many columns as will comfortably fit, type and spacing that scale smoothly with the viewport, sizes expressed as a range rather than a fixed number. The result adapts at every width, not just at five of them, and it needs far fewer media queries to do it.
  </p>

  <div className="rs-demo">
    <div className="rs-fluid-grid">
      <div className="rs-fluid-card"><span className="material-symbols-outlined">confirmation_number</span><span className="rs-fluid-card-title">Reflow</span><span className="rs-fluid-card-text">Cards wrap on their own as space allows.</span></div>
      <div className="rs-fluid-card"><span className="material-symbols-outlined">grid_view</span><span className="rs-fluid-card-title">auto-fit</span><span className="rs-fluid-card-text">Column count follows the container width.</span></div>
      <div className="rs-fluid-card"><span className="material-symbols-outlined">width</span><span className="rs-fluid-card-title">minmax</span><span className="rs-fluid-card-text">Each column stays between a floor and a fair share.</span></div>
      <div className="rs-fluid-card"><span className="material-symbols-outlined">done_all</span><span className="rs-fluid-card-title">Zero queries</span><span className="rs-fluid-card-text">No breakpoint wrote this. The grid did.</span></div>
    </div>
    <p className="rs-demo-caption">
      Live example. One rule, no media query: <code>grid-template-columns: repeat(auto-fit, minmax(160px, 1fr))</code>. Narrow the docs panel and the columns reflow on their own. This is the first tool to reach for, before any breakpoint.
    </p>
  </div>

  <div className="rs-demo" style={{ marginTop: '1.25rem' }}>
    <p className="rs-fluid-title">Fluid type scales with the screen</p>
    <p className="rs-demo-caption">
      This heading uses <code>clamp(1.5rem, 3vw + 1rem, 2.75rem)</code>. It grows and shrinks smoothly with the viewport, held between a minimum and a maximum so it never gets too small to read or too large to sit comfortably. Use <code>clamp()</code>, <code>min()</code>, and <code>max()</code> for type, spacing, and measure to erase the small step changes a stack of breakpoints would otherwise cause.
    </p>
  </div>
</div>

<div className="rs-section">
  <h2 className="rs-section-title"><span className="material-symbols-outlined">crop_din</span>Safe Areas and Notches</h2>
  <p className="rs-lede">
    On modern phones the screen is not a clean rectangle. Notches, rounded corners, and the home indicator carve into it, and a layout that draws all the way to the physical edge can hide a primary action underneath a system gesture zone. The browser exposes these insets through the <code>env(safe-area-inset-&#42;)</code> environment variables, unlocked by <code>viewport-fit=cover</code> on the viewport meta tag. Honor them for anything anchored to a screen edge: sticky bars, floating buttons, bottom-fixed CTAs. Add the inset to your spacing token rather than replacing it, so the padding is correct on a phone with a notch and unchanged on a screen without one.
  </p>

  <div className="rs-safearea">
    <div className="rs-phone" aria-hidden="true">
      <div className="rs-phone-notch"></div>
      <div className="rs-phone-safe">safe area</div>
      <div className="rs-phone-cta">Get tickets</div>
      <div className="rs-phone-indicator"></div>
    </div>
    <div>
      <p className="rs-lede" style={{ marginBottom: '0.75rem' }}>
        The dashed inset is the safe area. The primary button sits above the home indicator because its padding is <code>calc(var(--space-4) + env(safe-area-inset-bottom))</code>, not a bare token. On a device with no inset, <code>env()</code> resolves to zero and the button keeps its normal spacing.
      </p>
      <p className="rs-lede" style={{ margin: 0, fontSize: '0.8125rem', color: '#64748b' }}>
        Spacing and thumb-reach ergonomics for these anchored elements live in Foundations / Spacing. This page only claims the responsibility: never let a screen-anchored control fall under system chrome.
      </p>
    </div>
  </div>
</div>

<div className="rs-section">
  <h2 className="rs-section-title"><span className="material-symbols-outlined">touch_app</span>Adapt to the Input, Not Just the Size</h2>
  <p className="rs-lede">
    Screen width is only half of responsive. The other half is how the person is touching the interface. A wide screen might be a laptop with a precise mouse or a tablet with a blunt fingertip, and those want different hit areas and different feedback. CSS answers this with interaction media features, <code>pointer</code>, <code>hover</code>, <code>any-pointer</code>, and <code>any-hover</code>, which describe the input rather than the size. Never assume a large viewport means a mouse. The full model for state by input modality, including why hover must never be the only signal, lives in <strong>Patterns / Interactive</strong>.
  </p>
</div>

<div className="rs-section">
  <h2 className="rs-section-title"><span className="material-symbols-outlined">rule</span>Rules of the System</h2>

  <div className="rs-rules-grid">
    <div className="rs-rules-col do">
      <div className="rs-rules-col-title"><span className="rs-rules-badge">+</span> Do</div>
      <ul className="rs-rules-list">
        <li>Write the base layout mobile first, with no media query, and enhance upward from there.</li>
        <li>Reach for fluid sizing first: <code>clamp()</code>, <code>min()</code>, <code>max()</code>, and <code>auto-fit</code> grids.</li>
        <li>Use container queries for any component that must work in more than one slot.</li>
        <li>Keep media queries for the page shell and for capability or preference queries.</li>
        <li>Key every media query to one of the five breakpoint tokens.</li>
        <li>Add <code>env(safe-area-inset-&#42;)</code> to anything anchored to a screen edge.</li>
      </ul>
    </div>
    <div className="rs-rules-col dont">
      <div className="rs-rules-col-title"><span className="rs-rules-badge">x</span> Don't</div>
      <ul className="rs-rules-list">
        <li>Design wide first and squeeze down to fit a phone.</li>
        <li>Add a breakpoint at a device width when nothing in the content is actually breaking.</li>
        <li>Invent a sixth breakpoint, an <code>xs</code>, or a one-off pixel value outside the token set.</li>
        <li>Use a viewport media query to lay out a reusable component that a container query should own.</li>
        <li>Assume a wide viewport means a mouse, or a narrow one means touch.</li>
        <li>Let a sticky bar or bottom CTA sit under the notch or home indicator.</li>
      </ul>
    </div>
  </div>
</div>

<div className="rs-section">
  <h2 className="rs-section-title"><span className="material-symbols-outlined">account_tree</span>Which Tool, in Order</h2>
  <p className="rs-lede">
    Responsiveness is a ladder, not a switch. Start at the top and stop at the first rung that solves the problem. Most layout adapts long before you reach a breakpoint.
  </p>

  <div className="rs-decision">
    <div className="rs-decision-step">
      <div className="rs-decision-text">Can the element simply <strong>flow</strong>? Reach for <strong>intrinsic layout</strong> first: an <code>auto-fit</code> grid with <code>minmax()</code>, flex with wrapping, or a <code>clamp()</code> size. No breakpoint needed and it adapts at every width.</div>
    </div>
    <div className="rs-decision-step">
      <div className="rs-decision-text">Does a <strong>reusable component</strong> need to change based on the space it was given, not the page? Give its wrapper a <strong>container</strong> and write a <code>@container</code> query. Now it is portable across every slot.</div>
    </div>
    <div className="rs-decision-step">
      <div className="rs-decision-text">Is this a change to the <strong>page shell as a whole</strong>, the global grid, navigation, or where the sidebar lives? Use a <strong>media query</strong> keyed to a breakpoint token, for example <code>@media (min-width: 992px)</code>.</div>
    </div>
    <div className="rs-decision-step">
      <div className="rs-decision-text">Is it about a <strong>capability or preference</strong> rather than size, dark mode, reduced motion, a coarse pointer? Use the matching <strong>media feature</strong>: <code>prefers-color-scheme</code>, <code>prefers-reduced-motion</code>, <code>pointer</code>.</div>
    </div>
  </div>
</div>
