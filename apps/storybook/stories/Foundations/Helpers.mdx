<div className="helpers-header">
  <h1 className="helpers-title">The Stagehands</h1>
  <p className="helpers-description">
    Behind every performance there is a crew making the show run. Utility classes are the stagehands of the system: small, single-purpose classes that do exactly one thing, invisible to the audience but essential to the act. A helper aligns a ticket counter, spaces two cards, or hides a label from sight while keeping it for screen readers. This page is not a dump of every class we could invent. It is a curated set with a rule for when each one earns its place.
  </p>
</div>

<div className="helpers-section">
  <h2 className="helpers-section-title">Token, Utility, or Component</h2>
  <p className="helpers-lede">
    Three layers carry styling in this system, and choosing the right one is most of the craft. A <strong>token</strong> is a named value, the single source of truth. A <strong>utility</strong> is a one-declaration class that applies a token to an element in markup. A <strong>component</strong> is a named, reusable pattern with its own scoped styles. Utilities sit deliberately in the middle: closer to the metal than a component, one level up from writing the raw value. Reach for the layer that answers the question you actually have.
  </p>

  <div className="triad">
    <div className="triad-card">
      <span className="triad-icon" aria-hidden="true">data_object</span>
      <div className="triad-name">Token</div>
      <div className="triad-role">The value itself, named once. <code>--space-4</code>, <code>--color-primary</code>, <code>--radius-lg</code>. Everything else refers back to these.</div>
      <span className="triad-q">Answers: what is the value?</span>
    </div>
    <div className="triad-card util">
      <span className="triad-icon" aria-hidden="true">bolt</span>
      <div className="triad-name">Utility</div>
      <div className="triad-role">One class, one declaration, sourced from a token. <code>.gap-4</code> sets <code>gap: var(--space-4)</code>. A quick, local adjustment in markup.</div>
      <span className="triad-q">Answers: apply one value, here, now.</span>
    </div>
    <div className="triad-card">
      <span className="triad-icon" aria-hidden="true">widgets</span>
      <div className="triad-name">Component</div>
      <div className="triad-role">A named pattern with scoped styles, structure, and states. An event card, a button, a checkout row. The main act.</div>
      <span className="triad-q">Answers: a whole reusable thing.</span>
    </div>
  </div>

  <div className="helpers-note">
    <span className="helpers-note-icon" aria-hidden="true">info</span>
    <div>
      <div className="helpers-note-title">The one boundary that keeps the layers honest</div>
      <div className="helpers-note-body">
        A utility never invents a value. It reads a token. <code>.gap-4</code> means <code>var(--space-4)</code>, not <code>16px</code> typed by hand. This is what separates a design-system utility from a loose grab bag of CSS: every helper is a thin, named shortcut to a decision the tokens already made. If a helper would hardcode a raw value, it does not belong here.
      </div>
    </div>
  </div>
</div>

<div className="helpers-section">
  <h2 className="helpers-section-title">When a Utility Is the Right Tool</h2>
  <p className="helpers-lede">
    Utilities shine for the small, structural, endlessly repeated adjustments that would be silly to name as components: a flex row, a gap, an alignment, a width, a text size. They keep those decisions in the markup where you can read the layout at a glance, and they guarantee the value came from a token. Utilities stop being the right tool the moment the markup starts carrying a long chain of them that always travels together. That chain is a component asking to be born. The methodology this system follows, Andy Bell's CUBE CSS, is explicit about this balance: composition and light utilities handle the layout scaffolding, and blocks (components) own the parts that repeat with meaning. Utilities are a scalpel, not the whole surgery.
  </p>
</div>

<div className="helpers-section">
  <h2 className="helpers-section-title">Inside .sr-only, Declaration by Declaration</h2>
  <p className="helpers-lede">
    Visually hidden content is a solved problem with a canonical recipe, refined over years by the accessibility community and shipped almost identically in HTML5 Boilerplate, Bootstrap, and Tailwind. It is worth understanding rather than copying blind, because the common shortcut of <code>display: none</code> or <code>visibility: hidden</code> also removes the element from the accessibility tree, which is the opposite of the goal. The recipe below hides the pixels while keeping the text available to a screen reader.
  </p>

  <div className="anatomy">
    <div className="anatomy-card">
      <span className="anatomy-decl">position: absolute</span>
      <p className="anatomy-text">Pulls the element out of layout flow so its collapsed box does not disturb anything around it.</p>
    </div>
    <div className="anatomy-card">
      <span className="anatomy-decl">width: 1px; height: 1px</span>
      <p className="anatomy-text">Collapses the box to a single pixel. Not zero, because a truly zero-sized box can be dropped from the tree.</p>
    </div>
    <div className="anatomy-card">
      <span className="anatomy-decl">overflow: hidden</span>
      <p className="anatomy-text">Clips the real content down to that one-pixel box so none of it paints.</p>
    </div>
    <div className="anatomy-card">
      <span className="anatomy-decl">clip: rect(0, 0, 0, 0)</span>
      <p className="anatomy-text">Clips the visible region to nothing, the historical belt-and-braces companion to overflow.</p>
    </div>
    <div className="anatomy-card">
      <span className="anatomy-decl">white-space: nowrap</span>
      <p className="anatomy-text">Prevents the collapsed text from wrapping, which can otherwise reintroduce a sliver of visible layout.</p>
    </div>
    <div className="anatomy-card">
      <span className="anatomy-decl">margin: -1px; padding: 0; border: 0</span>
      <p className="anatomy-text">Zeroes the box model and nudges it off-screen so nothing leaks, not a border, not a background.</p>
    </div>
  </div>

  <div className="helpers-code">
    <div className="helpers-code-header">
      <span className="helpers-code-dot red"></span>
      <span className="helpers-code-dot yellow"></span>
      <span className="helpers-code-dot green"></span>

<div className="helpers-section">
  <h2 className="helpers-section-title">Rules of the Set</h2>
  <div className="rules-grid">
    <div className="rules-col do">
      <div className="rules-col-title"><span className="rules-badge">+</span> Do</div>
      <ul className="rules-list">
        <li>Keep every utility to a single declaration sourced from a token.</li>
        <li>Name utilities after the token they carry, so <code>--radius-pill</code> becomes <code>.rounded-pill</code>.</li>
        <li>Use helpers for local, structural tweaks: a gap, an alignment, a width.</li>
        <li>Treat <code>.sr-only</code> and <code>.focus-ring</code> as required accessibility infrastructure.</li>
        <li>Promote a stack of utilities to a component the moment it repeats with meaning.</li>
        <li>Prefer a native <code>&lt;button&gt;</code> or <code>&lt;a&gt;</code> over a <code>.cursor-pointer</code> div.</li>
      </ul>
    </div>
    <div className="rules-col dont">
      <div className="rules-col-title"><span className="rules-badge">x</span> Don't</div>
      <ul className="rules-list">
        <li>Add a utility that hardcodes a raw value instead of a token.</li>
        <li>Invent a helper that only re-expresses a token you already ship.</li>
        <li>Let a long, fixed chain of utilities stand in for a component.</li>
        <li>Use <code>.sr-only</code> to hide real content or hoard invisible keywords.</li>
        <li>Reach for a utility to carry business logic. That lives in the component.</li>
        <li>Grow the set for its own sake. Every class must earn a spot.</li>
      </ul>
    </div>
  </div>
</div>

<div className="helpers-section">
  <h2 className="helpers-section-title">Which Layer, in Five Questions</h2>
  <div className="decision">
    <div className="decision-step">
      <div className="decision-text">Do you just need the <strong>value</strong>, inside a component's own scoped CSS? Reference the <strong>token</strong> directly, for example <code>var(--space-4)</code>. No utility needed.</div>
    </div>
    <div className="decision-step">
      <div className="decision-text">Is it one structural adjustment in markup, a gap, an alignment, a width, a text size? Use the matching <strong>utility</strong>.</div>
    </div>
    <div className="decision-step">
      <div className="decision-text">Are the same three or four utilities always appearing together on the same kind of element? That pattern has a name. Extract a <strong>component</strong>.</div>
    </div>
    <div className="decision-step">
      <div className="decision-text">Does the element need structure, states, or behavior, an event card, a modal, a checkout row? It was always a <strong>component</strong>. Do not assemble it from utilities.</div>
    </div>
    <div className="decision-step">
      <div className="decision-text">Is the need purely for assistive technology, hiding a label or showing focus? Use <strong><code>.sr-only</code></strong> or <strong><code>.focus-ring</code></strong>. This is infrastructure, not styling.</div>
    </div>
  </div>
</div>
