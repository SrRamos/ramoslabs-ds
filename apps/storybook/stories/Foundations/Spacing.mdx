<div className="sp-header">
  <div className="sp-eyebrow">Foundations</div>
  <h1 className="sp-title">Spacing: the invisible grid</h1>
  <p className="sp-lede">
    Spacing is the quietest decision in an interface and the one users feel most. A single systematic scale, built on one small base unit, turns a thousand arbitrary pixel choices into a handful of deliberate ones. Consistent space creates rhythm, groups what belongs together, and gives every layout room to breathe.
  </p>
</div>

<div className="sp-section">
  <h2 className="sp-section-title">Why a scale, not pixels</h2>
  <p className="sp-prose">
    When spacing is chosen ad hoc, values drift: a <span className="sp-inline-code">13px</span> here, a <span className="sp-inline-code">15px</span> there, gaps that almost match but never quite do. The eye reads the inconsistency as sloppiness even when it cannot name it. A spacing scale removes the guesswork by offering a <strong>constrained set of values that all share a common factor</strong>, so any two measurements in the system relate to each other cleanly.
  </p>
  <p className="sp-prose">
    Refactoring UI makes the case bluntly: pick values in advance and only use those, because when every measurement is a multiple of a base unit, spacing decisions become fast and the result stays visually coherent. Material Design encodes the same idea as a layout grid: components and spacing align to a repeating measure so that composition is predictable across every screen.
  </p>
  <div className="sp-callout">
    <div className="sp-callout-mark">i</div>
    <div className="sp-callout-body">
      <div className="sp-callout-title">The core principle</div>
      A good spacing system does not ask "how many pixels here?" It asks "how many steps of the base unit?" Fewer choices, more consistency.
    </div>
  </div>
</div>

<div className="sp-section">
  <h2 className="sp-section-title">The base grid: 4px unit, 8pt rhythm</h2>
  <p className="sp-prose">
    Our scale is built on a <strong>4px base unit</strong> with an <strong>8px primary rhythm</strong>. The base of 4 gives fine control for small, dense adjustments (icon gaps, input insets), while the dominant steps land on multiples of 8. Eight is the workhorse because most screen densities divide it cleanly, so an 8px measure renders crisply from a 1x phone to a 3x display without landing on a half pixel.
  </p>
  <p className="sp-prose">
    This is the widely adopted "8pt grid": Material Design aligns layout to an 8dp measure (with a 4dp sub-grid for iconography and small type), and the practice is documented across Refactoring UI and mainstream design-system guidance. The payoff is compounding, spacing, component sizes, and line boxes all snap to the same underlying beat.
  </p>

Every value is a multiple of the 4px base; from <span className="sp-inline-code">--space-6</span> upward the steps widen (skipping 7, 9, 11, and so on) because at larger sizes the eye needs a bigger jump to perceive a real difference. <div className="sp-section">
  <h2 className="sp-section-title">Applying space: padding, margin, gap</h2>
  <p className="sp-prose">
    The same token behaves differently depending on which property carries it. Choosing the right property is what keeps a layout robust as content changes.
  </p>
  <div className="sp-grid-3">
    <div className="sp-concept">
      <span className="sp-concept-tag">inside</span>
      <div className="sp-concept-name">Padding</div>
      <div className="sp-concept-body">Space <strong>within</strong> an element, between its border and its content. Owns the breathing room inside cards, buttons, and inputs. Padding belongs to the component and travels with it.</div>
    </div>
    <div className="sp-concept">
      <span className="sp-concept-tag">between</span>
      <div className="sp-concept-name">Margin</div>
      <div className="sp-concept-body">Space <strong>outside</strong> an element, pushing neighbors away. Use it sparingly and in one direction (prefer a single <span className="sp-inline-code">margin-bottom</span> flow) to avoid collapsing-margin surprises.</div>
    </div>
    <div className="sp-concept">
      <span className="sp-concept-tag">layout</span>
      <div className="sp-concept-name">Gap</div>
      <div className="sp-concept-body">Space <strong>distributed</strong> by a flex or grid container between its children. One declaration on the parent spaces every child evenly, with no leftover margin on the last item. Prefer <span className="sp-inline-code">gap</span> for lists, toolbars, and grids.</div>
    </div>
  </div>
  <p className="sp-prose" style={{ marginTop: '1.25rem' }}>
    <strong>Vertical rhythm.</strong> Stacked content reads best when the vertical gaps are themselves multiples of the base unit and relate to the type's line height. Keep the space between a heading and its body tighter than the space before the next heading, so proximity signals grouping. As a rule of thumb, the gap that separates two groups should be visibly larger than any gap inside a group.
  </p>
  <p className="sp-prose">
    <strong>Sections and components.</strong> Scale spacing up with the size of the boundary. Inside a component use the small steps (<span className="sp-inline-code">--space-2</span> to <span className="sp-inline-code">--space-4</span>); between components use the medium steps (<span className="sp-inline-code">--space-6</span> to <span className="sp-inline-code">--space-10</span>); between full page sections use the large steps (<span className="sp-inline-code">--space-16</span> and up). The jump in magnitude is what makes structure legible at a glance.
  </p>
</div>

<div className="sp-section">
  <h2 className="sp-section-title">Touch targets and mobile</h2>
  <p className="sp-prose">
    Spacing is also an accessibility contract. On touch screens, an element is only usable if the finger can hit it reliably. Three numbers anchor the system, and they are non-negotiable floors, not suggestions.
  </p>
  <div className="sp-target-row">
    <div className="sp-target">
      <div className="sp-target-box min" style={{ width: '48px', height: '48px' }}>24</div>
      <div className="sp-target-meta">
        <div className="sp-target-label">24 &times; 24px</div>
        <div className="sp-target-sub">WCAG 2.2 floor (AA)</div>
      </div>
    </div>
    <div className="sp-target">
      <div className="sp-target-box apple" style={{ width: '88px', height: '88px' }}>44</div>
      <div className="sp-target-meta">
        <div className="sp-target-label">44 &times; 44pt</div>
        <div className="sp-target-sub">Apple HIG target</div>
      </div>
    </div>
    <div className="sp-target">
      <div className="sp-target-box material" style={{ width: '96px', height: '96px' }}>48</div>
      <div className="sp-target-meta">
        <div className="sp-target-label">48 &times; 48dp</div>
        <div className="sp-target-sub">Material target</div>
      </div>
    </div>
  </div>
  <p className="sp-prose" style={{ marginTop: '1.25rem' }}>
    <strong>The floor.</strong> WCAG 2.2 Success Criterion 2.5.8, Target Size (Minimum), requires interactive targets to be at least <strong>24 by 24 CSS pixels</strong> at conformance level AA (with spacing exceptions when targets are smaller). This is the absolute minimum for a compliant interface.
  </p>
  <p className="sp-prose">
    <strong>The target.</strong> Platform guidance sets a more comfortable bar. Apple's Human Interface Guidelines recommend a minimum tappable area of <strong>44 by 44 points</strong>; Material Design specifies a minimum touch target of <strong>48 by 48 dp</strong>. Design primary actions to the 44 to 48 range and reserve the 24px floor only for dense, secondary controls that also satisfy the WCAG spacing exception. Note the units differ: WCAG counts CSS pixels, Apple counts points, Material counts density-independent pixels, but in practice all three converge near the same physical size.
  </p>
  <p className="sp-prose">
    <strong>Safe areas.</strong> On modern phones the usable rectangle is inset by notches, rounded corners, and the home indicator. Apple's layout guidance calls this the safe area; on the web, honor it with the CSS <span className="sp-inline-code">env(safe-area-inset-&#42;)</span> variables (paired with <span className="sp-inline-code">viewport-fit=cover</span>) so sticky bars and primary CTAs never sit under a system gesture zone. Add your scale token to the inset rather than replacing it: <span className="sp-inline-code">padding-bottom: calc(var(--space-4) + env(safe-area-inset-bottom))</span>.
  </p>
  <div className="sp-callout">
    <div className="sp-callout-mark">!</div>
    <div className="sp-callout-body">
      <div className="sp-callout-title">Hit area is not visual size</div>
      A 20px icon can still meet the target floor: expand its tappable area with padding or a pseudo-element so the finger target reaches 44 to 48px even when the glyph stays small.
    </div>
  </div>
</div>

<div className="sp-section">
  <h2 className="sp-section-title">Density: comfortable by default</h2>
  <p className="sp-prose">
    Density is how tightly the same components pack together. Material Design frames it as an adjustable property: a <strong>comfortable</strong> default suits touch and reading, while a <strong>compact</strong> mode fits more information on screen for expert, data-dense, pointer-driven contexts. Density is applied by reducing the spacing step, not by inventing new off-scale values.
  </p>
  <table className="sp-table">
    <thead>
      <tr>
        <th>Mode</th>
        <th>Interior step</th>
        <th>When to apply</th>
      </tr>
    </thead>
    <tbody>
      <tr>
        <td><strong>Comfortable</strong> (default)</td>
        <td><code>--space-4</code> / <code>--space-6</code></td>
        <td>Touch-first screens, marketing, reading, onboarding. The safe default for anything a first-time user touches.</td>
      </tr>
      <tr>
        <td><strong>Compact</strong></td>
        <td><code>--space-2</code> / <code>--space-3</code></td>
        <td>Data tables, dashboards, admin tools, and pointer-driven power-user views where scanning many rows matters more than roominess.</td>
      </tr>
    </tbody>
  </table>
  <p className="sp-prose" style={{ marginTop: '1rem' }}>
    Even in compact mode the touch-target floor still applies: shrink the padding, never the hit area below 24px. Density lowers visual whitespace, it does not lower accessibility.
  </p>
</div>

<div className="sp-section">
  <h2 className="sp-section-title">Taxonomy: rules for a new project</h2>
  <p className="sp-prose">
    When you start a project on this system, these rules turn the scale into consistent output. Follow them and spacing stops being a per-screen negotiation.
  </p>
  <div className="sp-rules">
    <div className="sp-rule-col">
      <div className="sp-rule-head do">
        <span className="sp-rule-mark do">&#10003;</span> Do
      </div>
      <ul className="sp-rule-list">
        <li>Use only values from the scale. Every gap resolves to a <code>--space-&#42;</code> token.</li>
        <li>Prefer <code>gap</code> on flex and grid parents over margins on children. One declaration, no orphan spacing.</li>
        <li>Flow margins in one direction (for example <code>margin-bottom</code> only) to avoid collapsing-margin bugs.</li>
        <li>Size interactive targets to 44 to 48px; keep 24px as the hard floor.</li>
        <li>Scale spacing up on larger viewports: small steps on mobile, larger section steps on desktop.</li>
        <li>Add safe-area insets to sticky and bottom-anchored elements with <code>env(safe-area-inset-&#42;)</code>.</li>
      </ul>
    </div>
    <div className="sp-rule-col">
      <div className="sp-rule-head dont">
        <span className="sp-rule-mark dont">&#10007;</span> Don't
      </div>
      <ul className="sp-rule-list">
        <li>Hardcode arbitrary pixels (<code>13px</code>, <code>15px</code>, <code>margin: 7px</code>). If it is not on the scale, it does not ship.</li>
        <li>Stack margins from both neighbors to fake a gap; use the parent's <code>gap</code> instead.</li>
        <li>Shrink a hit area below 24px to save space, even in compact density.</li>
        <li>Invent a one-off value because two scale steps feel almost right; pick the nearer step.</li>
        <li>Let a sticky CTA sit under the home indicator by ignoring safe-area insets.</li>
        <li>Mix numeric and alias tokens for the same measurement in one component.</li>
      </ul>
    </div>
  </div>
  <p className="sp-prose" style={{ marginTop: '1.5rem' }}>
    <strong>Decision ladder.</strong> Stop at the first step that answers the question.
  </p>
  <div className="sp-flow">
    <div className="sp-flow-step">
      <span className="sp-flow-num">1</span>
      <span className="sp-flow-text"><strong>Is this space inside, between, or distributed?</strong> Inside a component uses padding, between two elements uses margin or a parent gap, across many children uses <code>gap</code>.</span>
    </div>
    <div className="sp-flow-step">
      <span className="sp-flow-num">2</span>
      <span className="sp-flow-text"><strong>How large is the boundary?</strong> Within a component reach for <code>--space-2</code> to <code>--space-4</code>; between components <code>--space-6</code> to <code>--space-10</code>; between page sections <code>--space-16</code> and up.</span>
    </div>
    <div className="sp-flow-step">
      <span className="sp-flow-num">3</span>
      <span className="sp-flow-text"><strong>Is it interactive?</strong> If so, verify the hit area meets the 44 to 48px target (24px hard floor) before finalizing the padding.</span>
    </div>
    <div className="sp-flow-step">
      <span className="sp-flow-num">4</span>
      <span className="sp-flow-text"><strong>Is it anchored to a screen edge?</strong> Add the matching <code>env(safe-area-inset-&#42;)</code> to the chosen token so it clears system chrome.</span>
    </div>
  </div>
</div>
