    <p className="mf-rule-why" style={{ marginTop: '0.75rem' }}><span className="mf-tag why">Note</span> The active tab is marked with <code>aria-current="page"</code>, so a screen reader announces it as current, not by color alone. The FAB carries an <code>aria-label</code> because its glyph has no visible text.</p>
  </div>
</div>

{/* 5: SNACKBAR */}
<div className="mf-section">
  <div className="mf-section-header">
    <span className="mf-section-number">5</span>
    <h2 className="mf-section-title">Snackbar, feedback in the thumb band</h2>
  </div>

  <p className="mf-text">
    Material: snackbars "provide brief messages about app processes at the bottom of the screen" and "shouldn't interrupt the user experience." Only one shows at a time (a new one dismisses any previous), it may carry a single optional action such as undo or retry, and it is "the preferred mechanism for displaying feedback messages," with toast reserved for when a snackbar cannot be used. It replaces the desktop toast that lived in the top-right corner. On web: a region with <code>role="status"</code> (or <code>aria-live="polite"</code>) anchored to the bottom with <code>env(safe-area-inset-bottom)</code>; if it has an action, that action is a real <code>&lt;button&gt;</code> reachable by keyboard, and never dependent on a swipe (Section 6).
  </p>

  <div className="mf-demo">
    <div className="mf-demo-label">Live: a bottom snackbar with a single Undo action, role="status"</div>

      </div>
    </div>
    <p className="mf-rule-why" style={{ marginTop: '0.75rem' }}><span className="mf-tag why">Note</span> The message sits in a polite live region so it is announced without stealing focus. The Undo action is a keyboard-reachable button, the same action a swipe would have triggered.</p>
  </div>
</div>

{/* 6: GESTURES */}
<div className="mf-section">
  <div className="mf-section-header">
    <span className="mf-section-number">6</span>
    <h2 className="mf-section-title">Gestures, and the rule that never bends</h2>
  </div>

  <p className="mf-text">
    Swipe to delete, long press for a context menu, pull to refresh, drag on a sheet's snap points: all useful, all mobile-native. The non-negotiable rule: <strong>a gesture can never be the only path.</strong> There must always be an alternative that works with a single pointer and with a keyboard.
  </p>

  <div className="mf-rules">
    <div className="mf-rule">
      <p className="mf-rule-line"><span className="mf-tag rule">Rule</span> Any path-based or multipoint gesture has a single-pointer alternative.</p>
      <p className="mf-rule-why"><span className="mf-tag why">Why</span> WCAG 2.5.1 Pointer Gestures (Level A): functionality using such gestures "can be operated with a single pointer without a path-based gesture" unless essential. A tap, click, or long press qualifies. Swipe to delete needs a visible Delete button.</p>
    </div>
    <div className="mf-rule">
      <p className="mf-rule-line"><span className="mf-tag rule">Rule</span> Any dragging movement has a no-drag alternative.</p>
      <p className="mf-rule-why"><span className="mf-tag why">Why</span> WCAG 2.5.7 Dragging Movements (2.2, Level AA): achievable "by a single pointer without dragging," for example tapping buttons to move items or picking from a pop-up menu. The sheet's drag handle needs buttons or taps that expand, collapse, and close it.</p>
    </div>
    <div className="mf-rule">
      <p className="mf-rule-line"><span className="mf-tag rule">Rule</span> Long press replaces right click only alongside a visible overflow control.</p>
      <p className="mf-rule-why"><span className="mf-tag why">Why</span> Long press is neither discoverable nor keyboard-operable. The same action must live in a visible "more options" menu so every user can reach it.</p>
    </div>
    <div className="mf-rule">
      <p className="mf-rule-line"><span className="mf-tag rule">Rule</span> Custom pull to refresh ships with a visible Refresh button.</p>
      <p className="mf-rule-why"><span className="mf-tag why">Why</span> Native browser scroll and the OS pull-to-refresh fall under the user-agent exception, but a gesture your own content interprets does not. If you build it, give the same action an explicit control.</p>
    </div>
  </div>

  <div className="mf-demo">

{/* 7: SAFE AREA */}
<div className="mf-section">
  <div className="mf-section-header">
    <span className="mf-section-number">7</span>
    <h2 className="mf-section-title">Safe-area insets, notch and home indicator</h2>
  </div>

  <p className="mf-text">
    MDN defines <code>env(safe-area-inset-*)</code> as "the safe distance from the top, right, bottom, or left inset edge of the viewport, defining where it is safe to place content without risking it being cut off by the shape of a non-rectangular display." The four variables are <code>safe-area-inset-top</code>, <code>-right</code>, <code>-bottom</code>, and <code>-left</code>. They are <code>0</code> on a plain rectangular viewport and larger than zero around a notch or the home indicator, but only when the page opts in with <code>&lt;meta name="viewport" content="viewport-fit=cover"&gt;</code>.
  </p>

  <div className="mf-code">
    <div><span className="mf-c-com">{'/* MDN canonical sticky bottom bar */'}</span></div>
    <div>footer &#123;</div>
    <div>&nbsp;&nbsp;<span className="mf-c-prop">position</span>: <span className="mf-c-val">sticky</span>;</div>
    <div>&nbsp;&nbsp;<span className="mf-c-prop">bottom</span>: <span className="mf-c-val">0</span>;</div>
    <div>&nbsp;&nbsp;<span className="mf-c-prop">padding</span>: <span className="mf-c-val">1em 1em calc(1em + env(safe-area-inset-bottom, 0px))</span>;</div>
    <div>&#125;</div>
  </div>

  <div className="mf-rules">
    <div className="mf-rule">
      <p className="mf-rule-line"><span className="mf-tag rule">Rule</span> Never pad a fixed edge with a bare constant. Use <code>calc(base + env(safe-area-inset-*, 0px))</code>.</p>
      <p className="mf-rule-why"><span className="mf-tag why">Why</span> The <code>calc()</code> keeps your normal spacing on rectangular screens and adds only the device inset where one exists. The <code>0px</code> fallback keeps it correct on browsers that do not supply the variable. <code>env()</code> has been widely available since January 2020.</p>
    </div>
  </div>

  <div className="mf-callout">
    <span className="mf-material" aria-hidden="true">science</span>
    <p className="mf-callout-text">
      <strong>Out of scope, not yet normative: the virtual keyboard.</strong> Handling the on-screen keyboard resizing the viewport (the VirtualKeyboard API, <code>interactive-widget</code>, dynamic viewport units) is a live and shifting area we have not verified against a primary source for this page. iOS documents automatic keyboard avoidance for sheets as a platform behavior, but the web story is experimental. Treat it as a follow-up, not a rule, and confirm current support before relying on it.
    </p>
  </div>
</div>

{/* 8: ACTIVATION */}
<div className="mf-section">
  <div className="mf-section-header">
    <span className="mf-section-number">8</span>
    <h2 className="mf-section-title">Enable the patterns on touch, without sniffing</h2>
  </div>

  <p className="mf-text">
    Do not read <code>navigator.userAgent</code>. Detect the capability of the input and the size of the viewport, because that is what actually decides whether these patterns fit. MDN's <code>pointer</code> media feature tests the primary pointer's accuracy: <code>coarse</code> is "a pointing device of limited accuracy, such as a finger on a touchscreen," <code>fine</code> is "an accurate pointing device, such as a mouse," and <code>none</code> is no pointer at all. Use <code>pointer</code> for the primary device and <code>any-pointer</code> for any device present.
  </p>

  <div className="mf-code">
    <div><span className="mf-c-com">{'/* Capability, not user agent */'}</span></div>
    <div>@media <span className="mf-c-val">(pointer: coarse)</span> &#123;</div>
    <div>&nbsp;&nbsp;<span className="mf-c-com">{'/* larger targets, touch patterns, no hover reveal */'}</span></div>
    <div>&#125;</div>
    <div>@media <span className="mf-c-val">(pointer: fine)</span> &#123;</div>
    <div>&nbsp;&nbsp;<span className="mf-c-com">{'/* mouse density, hover affordances */'}</span></div>
    <div>&#125;</div>
    <div><span className="mf-c-com">{'/* Switch container by capability OR width */'}</span></div>
    <div>@media <span className="mf-c-val">(pointer: coarse), (max-width: 640px)</span> &#123; <span className="mf-c-com">{'/* sheet, bottom nav */'}</span> &#125;</div>
  </div>

  <div className="mf-rules">
    <div className="mf-rule">
      <p className="mf-rule-line"><span className="mf-tag rule">Rule</span> On <code>pointer: coarse</code>, assume no reliable hover. Reveal-on-hover patterns do not exist; make actions visible or reveal on tap.</p>
      <p className="mf-rule-why"><span className="mf-tag why">Why</span> A touchscreen has no persistent hover state. Anything hidden behind <code>:hover</code> is simply unreachable by a finger, which is why the map has no mobile home for "hover to reveal."</p>
    </div>
    <div className="mf-rule">
      <p className="mf-rule-line"><span className="mf-tag rule">Rule</span> Switch the container on <code>pointer: coarse</code> OR width at or below the breakpoint, so both phones and touch tablets are covered.</p>
      <p className="mf-rule-why"><span className="mf-tag why">Why</span> Width alone misses a small touch tablet; capability alone misses a narrow mouse window. The union of the two is what reliably picks the sheet-and-bottom-nav layout. <code>pointer</code> has been widely available since December 2018.</p>
    </div>
  </div>

  <div className="mf-chips">
    <span className="mf-chip">pointer: coarse<button type="button" className="mf-chip-x" aria-label="Remove filter, pointer coarse"><span className="mf-material" aria-hidden="true">close</span></button></span>
    <span className="mf-chip">width breakpoint<button type="button" className="mf-chip-x" aria-label="Remove filter, width breakpoint"><span className="mf-material" aria-hidden="true">close</span></button></span>
    <span className="mf-chip">any-hover: none<button type="button" className="mf-chip-x" aria-label="Remove filter, any-hover none"><span className="mf-material" aria-hidden="true">close</span></button></span>
  </div>
  <p className="mf-rule-why"><span className="mf-tag why">Why</span> These compact chips are the whole detection surface: primary pointer accuracy, viewport width, and hover availability. No user-agent string appears anywhere in it.</p>
</div>
