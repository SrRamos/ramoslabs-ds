<div className="dt-section">
  <div className="dt-section-header">
    <span className="dt-section-number">1</span>
    <h2 className="dt-section-title">Native &lt;table&gt; first, never div-soup</h2>
  </div>

  <p className="dt-text">
    Build the table from real elements: a <code>&lt;table&gt;</code> with a <code>&lt;caption&gt;</code>, a <code>&lt;thead&gt;</code> and <code>&lt;tbody&gt;</code>, and <code>&lt;th&gt;</code> cells that carry <code>scope</code>. This is not stylistic. Without <code>&lt;th&gt;</code>, <code>scope</code> and the table role, a screen reader cannot announce "column Amount, row Order 4521" when the user lands on a cell. The row and column relationship is simply lost, and there is no ARIA shortcut simpler than using the real element. MDN puts it plainly: a logical structure with semantic markup is not only easier to style, it is what makes a table understandable and navigable by assistive technology and search engines alike.
  </p>

  <div className="dt-rules">
    <div className="dt-rule">
      <p className="dt-rule-line"><span className="dt-tag rule">Rule</span> Use a real <code>&lt;table&gt;</code> with <code>&lt;caption&gt;</code>, <code>&lt;thead&gt;</code> / <code>&lt;tbody&gt;</code>, and <code>&lt;th scope="col"&gt;</code> for column headers plus <code>&lt;th scope="row"&gt;</code> for each row's identifying cell.</p>
      <p className="dt-rule-why"><span className="dt-tag why">Why</span> WCAG 2.1 SC 1.3.1 Info and Relationships (Level A) requires that structure conveyed visually be programmatically determinable. Scope is inferred in simple tables, but MDN and GOV.UK both advise setting it explicitly, because some assistive technology draws the wrong inference.</p>
    </div>
    <div className="dt-rule">
      <p className="dt-rule-line"><span className="dt-tag rule">Rule</span> The <code>&lt;caption&gt;</code> is the table's accessible name. Give every table one.</p>
      <p className="dt-rule-why"><span className="dt-tag why">Why</span> GOV.UK: a caption helps users find, navigate and understand tables. It lets a screen reader user decide whether to read on or skip the table entirely.</p>
    </div>
    <div className="dt-rule">
      <p className="dt-rule-line"><span className="dt-tag rule">Rule</span> For multi-level or spanned headers, prefer splitting into smaller related tables over reaching for <code>headers</code> / <code>id</code> wiring.</p>
      <p className="dt-rule-why"><span className="dt-tag why">Why</span> MDN recommends breaking a complex table apart rather than relying on <code>colspan</code> / <code>rowspan</code>. It helps assistive technology and lowers cognitive load. If you are reaching for <code>headers</code> / <code>id</code>, first ask whether this should be two tables.</p>
    </div>
  </div>

{/* 2: THE FORK */}
<div className="dt-section">
  <div className="dt-section-header">
    <span className="dt-section-number">2</span>
    <h2 className="dt-section-title">The fork: &lt;table&gt; vs role="grid"</h2>
  </div>

  <p className="dt-text">
    This is the single most misunderstood decision in table design, and the most common mistake is over-reaching for <code>role="grid"</code>. The default is a static <code>&lt;table&gt;</code>. Most product tables, even with sorting, filtering, pagination, row links and row actions, are <strong>not</strong> grids. The WAI-ARIA APG is explicit: like an HTML table, a WAI-ARIA table is a static tabular structure, not an interactive widget. You only graduate to a grid when the table is a spreadsheet the user operates cell by cell with arrow keys.
  </p>

  <div className="dt-ref-scroll">
    <table className="dt-fork">
      <thead>
        <tr>
          <th className="a">Use a static <code>&lt;table&gt;</code> (default) when</th>
          <th className="b">Use <code>role="grid"</code> only when</th>
        </tr>
      </thead>
      <tbody>
        <tr>
          <td className="a">The table presents data to read, scan and compare.</td>
          <td className="b">The table is an interactive widget operated like a spreadsheet.</td>
        </tr>
        <tr>
          <td className="a">Cells may hold links or buttons and the user tabs through them normally.</td>
          <td className="b">You need cell-by-cell arrow-key navigation with a single tab stop for the whole grid.</td>
        </tr>
        <tr>
          <td className="a">You have sorting, filtering, pagination. None of these require a grid.</td>
          <td className="b">You have editable cells or complex keyboard row and column selection.</td>
        </tr>
      </tbody>
    </table>
  </div>

  <div className="dt-callout">
    <span className="dt-material" aria-hidden="true">warning</span>
    <p className="dt-callout-text">
      <strong>A grid is a heavyweight commitment.</strong> Once you adopt <code>role="grid"</code> you own focus management by roving tabindex (exactly one focusable element in the tab sequence at a time) and the full keyboard model: arrow keys move one cell and stop at edges, <code>Home</code> / <code>End</code> jump to row ends, <code>Ctrl+Home</code> / <code>Ctrl+End</code> to grid corners, and in an edit grid <code>Enter</code> or <code>F2</code> enters edit mode while <code>Escape</code> returns to navigation. Do not sign up for this to get sortable columns.
    </p>
  </div>
</div>

<div className="dt-section">

  <div className="dt-rules">
    <div className="dt-rule">
      <p className="dt-rule-line"><span className="dt-tag rule">Rule</span> The sort control is a native <code>&lt;button&gt;</code> inside the <code>&lt;th&gt;</code>. Sort state lives in <code>aria-sort</code> on that <code>&lt;th&gt;</code>, on one column at a time.</p>

    </div>
    <div className="dt-rule">

<div className="dt-section">

    <h2 className="dt-section-title">Mobile, the hard problem</h2>
  </div>

  <p className="dt-text">
    Tables do not fit narrow viewports, and WCAG SC 1.4.10 Reflow gives them a specific carve-out. Content must reflow to 320 CSS px without two-dimensional scrolling, <strong>except for parts that require a two-dimensional layout for meaning</strong>, and the standard names "data tables (not individual cells)" as exactly such a part. In plain terms: a data table is permitted to scroll horizontally on a phone and you do not fail Reflow for it, as long as the surrounding page (headings, search, filters) still reflows. Horizontal scroll is a legitimate, standards-blessed strategy for tables, not a hack.
  </p>

      <p className="dt-pattern-body">Keep the real <code>&lt;table&gt;</code>. Wrap it in a focusable, labeled scroll region and freeze the identifying first column as the anchor.</p>
      <p className="dt-pattern-tradeoff"><b>Gain:</b> bulletproof semantics, about six lines of CSS, no JS, no data duplication. <i>Lose:</i> the user scrolls to reach far columns, so mitigate with the sticky first column.</p>
    </div>
    <div className="dt-pattern">
      <div className="dt-pattern-head">
        <span className="dt-pattern-name">B. Stacked cards</span>
        <span className="dt-verdict care">Repair needed</span>
      </div>
      <p className="dt-pattern-body">Collapse each row into a key and value card with <code>td::before</code> labels. Readable one record at a time.</p>
      <p className="dt-pattern-tradeoff"><b>Gain:</b> no horizontal scroll, great for inspecting single records. <i>Lose:</i> <code>display:block</code> destroys table semantics, so you MUST restore roles with ARIA, and cross-record comparison becomes nearly impossible.</p>
    </div>
    <div className="dt-pattern">

        <span className="dt-pattern-name">C. Priority columns + expand</span>

    </div>

        <span className="dt-pattern-name">D. Not a table at all</span>

We trust Roselli, the AT-tested specialist. If you use the card pattern you must repair semantics with ARIA roles. Otherwise, prefer Pattern A.
