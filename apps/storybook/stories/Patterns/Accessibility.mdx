  <div className="a11y-section-header">

  </div>

  <p className="a11y-section-text">
    The order of operations matters. Reach for the native element first: <code>&lt;button&gt;</code> over <code>&lt;div onclick&gt;</code>, <code>&lt;nav&gt;</code> over <code>&lt;div class="nav"&gt;</code>, <code>&lt;a href&gt;</code> for anything that navigates. Native elements ship with a role, a name, keyboard behavior, and focus management already wired in. ARIA adds none of that on its own: it changes what assistive technology announces, but you still have to build the keyboard support and state by hand<span className="a11y-ref">[3]</span>. This is why the first rule of ARIA is to not use ARIA.
  </p>

  <p className="a11y-section-text" style={{ marginTop: '1rem' }}>
    The <strong>five rules of ARIA</strong><span className="a11y-ref">[4]</span> keep that discipline honest:
  </p>

  <ol className="a11y-numbered">
    <li>If a native HTML element or attribute already gives you the semantics and behavior you need, use it instead of repurposing an element and adding an ARIA role. A real <code>&lt;button&gt;</code> beats <code>&lt;div role="button" tabindex="0"&gt;</code> every time.</li>
    <li>Do not change native semantics unless you truly have to. Wrapping a heading in <code>role="tab"</code> hides the heading from users who navigate by headings. Nest a real element inside instead.</li>
    <li>Every interactive ARIA control must be operable from the keyboard. If you add <code>role="button"</code>, you own <code>Enter</code>, <code>Space</code>, focus, and the visible focus ring.</li>
    <li>Never put <code>role="presentation"</code> or <code>aria-hidden="true"</code> on a focusable element. A control that is focusable but hidden from the accessibility tree strands screen reader users on nothing.</li>
    <li>Every interactive element needs an accessible name. A visible <code>&lt;label&gt;</code>, the element's text, <code>aria-label</code>, or <code>aria-labelledby</code>, in that order of preference.</li>
  </ol>

  <p className="a11y-section-text">
    The takeaway is blunt: <strong>no ARIA is better than bad ARIA</strong><span className="a11y-ref">[4]</span>. If you are unsure, drop to semantic HTML and you will almost always be more accessible, not less.
  </p>
</div>

<div className="a11y-section">
  <div className="a11y-section-header">
    <span className="a11y-section-number">4</span>
    <h2 className="a11y-section-title">Keyboard and Focus</h2>
  </div>

  <p className="a11y-section-text">
    Many users navigate entirely with a keyboard, whether from a motor impairment, a repetitive strain injury, a screen reader, or plain preference. <strong>WCAG 2.2 SC 2.1.1</strong> means every interactive element is reachable with <strong>Tab</strong>, activatable with <strong>Enter</strong> or <strong>Space</strong>, and dismissable with <strong>Escape</strong> where a layer is open<span className="a11y-ref">[8]</span>. And focus can never be a trap: if <code>Tab</code> takes you in, <code>Tab</code> or <code>Escape</code> must take you out.

  <p className="a11y-section-text" style={{ marginTop: '1rem' }}>
    Focus has to be <strong>visible</strong> (<strong>SC 2.4.7</strong>), and WCAG 2.2 raises the bar: the focused control must not be hidden behind sticky headers or toolbars (<strong>SC 2.4.11 Focus Not Obscured</strong>), and the focus indicator itself needs enough area and contrast to be seen (<strong>SC 2.4.13 Focus Appearance</strong>)<span className="a11y-ref">[9]</span>.     <h2 className="a11y-section-title">Target Size and Motion</h2>
  </div>

  <p className="a11y-section-text">
    A control has to be big enough to hit. <strong>WCAG 2.2 SC 2.5.8</strong> sets a floor of <strong>24 by 24 CSS pixels</strong> at Level AA, with spacing exceptions for inline links<span className="a11y-ref">[10]</span>. When a control looks smaller, the <code>.touch-target</code> helper pads its hit area without changing the visual.
  </p>

Honor <code>prefers-reduced-motion</code> as a floor: collapse the motion, keep the meaning. The <code>.no-motion</code> helper is the manual switch for a JS toggle.
  </p>

{/* CHAPTER 6: SCREEN READERS, NAME/ROLE/VALUE   */}
{/* ============================================ */}

<div className="a11y-section">
  <div className="a11y-section-header">
    <span className="a11y-section-number">6</span>
    <h2 className="a11y-section-title">Screen Readers: Name, Role, and Value</h2>
  </div>

  <p className="a11y-section-text">
    A screen reader reads three things off every control, and <strong>WCAG 2.2 SC 4.1.2</strong> requires all three<span className="a11y-ref">[13]</span>. The <strong>name</strong> is what it is called ("Search"). The <strong>role</strong> is what kind of thing it is (button, checkbox, tab). The <strong>value</strong> or state is where it stands right now (checked, expanded, "3 of 10"). A native <code>&lt;input type="checkbox"&gt;</code> with a label supplies all three for free. A <code>&lt;div&gt;</code> styled to look like a checkbox supplies none, so a custom widget must set <code>role</code>, an accessible name, and <code>aria-checked</code> by hand and keep the value in sync.
  </p>

  <p className="a11y-section-text" style={{ marginTop: '1rem' }}>
    <strong>Live regions</strong> announce changes that happen away from focus: a cart count ticking up, a validation error, search results loading. Use <code>role="alert"</code> (assertive) for urgent messages that should interrupt, and <code>role="status"</code> or <code>aria-live="polite"</code> for updates that can wait for a pause<span className="a11y-ref">[14]</span>. Add <code>aria-atomic="true"</code> when the whole message must be read, not just the changed word.
  </p>
