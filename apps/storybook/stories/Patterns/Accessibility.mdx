{/* ===== 2. SEMANTIC HTML BEFORE ARIA ===== */}

<div className="a11y-section">
  <div className="a11y-section-header">
    <span className="a11y-section-number">2</span>
    <h2 className="a11y-section-title">Semantic HTML Before ARIA</h2>
  </div>

  <p className="a11y-section-text">
    The first rule of ARIA is: do not use ARIA. Reach for the native element first, because it hands you the accessibility that a styled <code>&lt;div&gt;</code> makes you rebuild from scratch.
  </p>

  <div className="a11y-rule">
    <div className="a11y-rule-label"><span className="material-symbols-outlined" aria-hidden="true">gavel</span>Rule</div>
    <div className="a11y-rule-text">Reach for the native element first, and add ARIA only to fill a gap it leaves.</div>
    <div className="a11y-rule-why"><strong>Why:</strong> a native <code>&lt;button&gt;</code>, <code>&lt;nav&gt;</code>, or <code>&lt;a href&gt;</code> ships with a role, an accessible name, keyboard behavior, and focus already wired in. ARIA changes only what a screen reader announces, so anything you build on a bare <code>&lt;div&gt;</code> you also have to make operable and focusable by hand.</div>
  </div>

  <div className="a11y-ra">
    <div className="a11y-ra-card rec">
      <div className="a11y-ra-head"><span className="material-symbols-outlined" aria-hidden="true">check_circle</span>Recommended</div>
      <code className="a11y-ra-code">&lt;button type="button"&gt;Buy tickets&lt;/button&gt;</code>
      <p className="a11y-ra-note">Role, accessible name, focus stop, and keyboard activation come for free, and it behaves the same in every browser and assistive tech.</p>
    </div>
    <div className="a11y-ra-card avoid">
      <div className="a11y-ra-head"><span className="material-symbols-outlined" aria-hidden="true">cancel</span>Avoid</div>
      <code className="a11y-ra-code">&lt;div onclick="buy()"&gt;Buy tickets&lt;/div&gt;</code>
      <p className="a11y-ra-note">No role, no keyboard, no focus stop. You would have to rebuild <code>tabindex</code>, key handling, and the focus ring by hand, and it is easy to miss one.</p>
    </div>
  </div>

  <ol className="a11y-numbered">
    <li>Prefer the native element whenever it carries the semantics and behavior you need. A real <code>&lt;button&gt;</code> beats <code>&lt;div role="button" tabindex="0"&gt;</code>.</li>
    <li>Keep native semantics intact: nest a real element inside rather than repainting one with a role.</li>
    <li>Keep every interactive ARIA control keyboard operable. Add <code>role="button"</code> and you own <code>Enter</code>, <code>Space</code>, focus, and the ring.</li>
    <li>Keep focusable elements visible to assistive tech: <code>aria-hidden="true"</code> and <code>role="presentation"</code> belong only on content that cannot receive focus.</li>
    <li>Give every interactive element an accessible name: a visible <code>&lt;label&gt;</code>, its text, <code>aria-label</code>, or <code>aria-labelledby</code>.</li>
  </ol>

  <p className="a11y-section-text">
    <strong>No ARIA is better than bad ARIA.</strong> Unsure? Drop to semantic HTML and you will almost always be more accessible, not less.
  </p>
</div>
