  <p className="fe-text">
    <strong>The label is always visible, above the field, never inside it.</strong> The tempting shortcut is to delete the label and let a placeholder stand in for it. It looks clean and it breaks the moment the user starts typing: the question vanishes exactly when they need to check what was asked, people with memory or attention differences lose the prompt entirely, and screen readers treat placeholder text as unreliable. <a className="fe-cite" href="https://www.nngroup.com/articles/form-design-placeholders/" target="_blank" rel="noreferrer">NN/g</a> is unambiguous that placeholders are not a substitute for labels. A persistent top-aligned label is also the fastest layout to scan, because the eye travels straight down a single column from question to answer, which is why <a className="fe-cite" href="https://baymard.com/blog/avoid-multi-column-forms" target="_blank" rel="noreferrer">Baymard</a> finds single-column forms are completed faster and with fewer skipped fields than multicolumn ones.

The type tells the operating system which on-screen keyboard to raise, so <code>type="email"</code> surfaces the @ key, <code>type="tel"</code> raises the big numeric dial pad, and <code>type="url"</code> puts <em>.com</em> a tap away. Choosing the wrong type does not just look wrong, it forces the user to hunt across keyboard layouts on every single field.
  </p>

  <p className="fe-text">
    Two attributes refine it further. <a className="fe-cite" href="https://developer.mozilla.org/en-US/docs/Web/HTML/Reference/Global_attributes/inputmode" target="_blank" rel="noreferrer"><code>inputmode</code></a> tunes the keyboard without changing the field's semantics, so <code>inputmode="numeric"</code> gives a digit pad to a one-time code that is still logically text. And <a className="fe-cite" href="https://developer.mozilla.org/en-US/docs/Web/HTML/Reference/Attributes/autocomplete" target="_blank" rel="noreferrer"><code>autocomplete</code></a> lets the browser fill the field from the user's saved profile, which is the difference between typing a full address and tapping it in once. Google's <a className="fe-cite" href="https://web.dev/articles/payment-and-address-form-best-practices" target="_blank" rel="noreferrer">form best-practice guidance</a> treats correct autocomplete tokens as table stakes for checkout.
  </p>

{/* CHAPTER 3: VALIDATION                          */}
{/* ============================================ */}

<div className="fe-section">
  <div className="fe-section-header">
    <span className="fe-section-number">3</span>
    <h2 className="fe-section-title">Validate at the Right Moment, in the Right Words</h2>
  </div>

  <p className="fe-text">
    Validation has two questions: <strong>when</strong> to check, and <strong>what</strong> to say. Get the timing wrong and you either nag someone mid-keystroke or ambush them at submit. The reliable rule, from the classic <a className="fe-cite" href="https://www.smashingmagazine.com/2022/09/inline-validation-web-forms-ux/" target="_blank" rel="noreferrer">inline-validation study by Mihael Konjević</a>, is <strong>reward early, punish late</strong>: confirm a field as valid while the user types if you can, but do not show an error until they have left the field. Validating on every keystroke means the first character of an email is flagged as a broken email, which is technically true and deeply unhelpful.
  </p>

  <div className="fe-timeline">
    <div className="fe-tl-step">
      <span className="fe-tl-when"><span className="fe-material" style={{ fontSize: '0.85rem' }} aria-hidden="true">keyboard</span> While typing</span>
      <div className="fe-tl-title">Reward, never punish</div>
      <div className="fe-tl-body">Show a success tick the moment input becomes valid. Hold every error back, because an incomplete field is not yet a wrong one.</div>
    </div>
    <div className="fe-tl-step">
      <span className="fe-tl-when"><span className="fe-material" style={{ fontSize: '0.85rem' }} aria-hidden="true">logout</span> On blur</span>
      <div className="fe-tl-title">Now you may correct</div>
      <div className="fe-tl-body">When focus leaves the field, the answer is final enough to judge. This is the moment to surface an inline error, right under the field.</div>
    </div>
    <div className="fe-tl-step">
      <span className="fe-tl-when"><span className="fe-material" style={{ fontSize: '0.85rem' }} aria-hidden="true">send</span> On submit</span>
      <div className="fe-tl-title">Catch the rest</div>
      <div className="fe-tl-body">Re-check everything, move focus to the first error, and summarise. Never let a form fail silently or scroll the user nowhere.</div>
    </div>
  </div>

  <p className="fe-text">
    A good error message does three things at once, per <a className="fe-cite" href="https://www.nngroup.com/articles/error-message-guidelines/" target="_blank" rel="noreferrer">NN/g's error-message guidelines</a>: it is <strong>visible and specific</strong> (next to the field, not a generic banner), it says <strong>what went wrong in plain language</strong>, and it tells the user <strong>how to fix it</strong>. "Invalid input" fails all three. "Enter a date after today, the event cannot start in the past" passes all three. And because <a className="fe-cite" href="https://www.w3.org/WAI/WCAG22/Understanding/error-identification.html" target="_blank" rel="noreferrer">WCAG 3.3.1</a> requires errors to be identified in text, never lean on a red border alone. The colour is an enhancement, the words are the message.
  </p>

  <div className="fe-lab">
    <div className="fe-demo">
      <div className="fe-demo-label">Reward: valid while typing</div>
      <div className="fe-field">
        <label className="fe-label" htmlFor="fe-ok">Email address</label>
        <input id="fe-ok" className="fe-input success" type="email" defaultValue="ana@ramoslabs.co" aria-describedby="fe-ok-msg" />
        <span id="fe-ok-msg" className="fe-success-text"><span className="fe-material" aria-hidden="true">check_circle</span> Looks good.</span>
      </div>
    </div>
    <div className="fe-demo">
      <div className="fe-demo-label">Punish late: actionable error on blur</div>
      <div className="fe-field">
        <label className="fe-label" htmlFor="fe-bad">Email address</label>
        <input id="fe-bad" className="fe-input error" type="email" defaultValue="ana@ramoslabs" aria-invalid="true" aria-describedby="fe-bad-msg" />
        <span id="fe-bad-msg" className="fe-error"><span className="fe-material" aria-hidden="true">error</span> Add the domain ending, like .co or .com, so the address can receive mail.</span>
      </div>
    </div>
  </div>

  <div className="fe-callout">
    <span className="fe-material" aria-hidden="true">palette</span>
    <p className="fe-callout-text">
      <strong>Error colour is <code>--color-error</code>, and it is the only warm colour on the page.</strong> Everything interactive stays indigo; the single job of red here is to mark a genuine problem. Do not tint help text, borders or icons red for emphasis. Reserve it, so that when it appears the user knows it means something is wrong.
    </p>
  </div>

{/* CHAPTER 4: FIELD STATES                        */}
{/* ============================================ */}

<div className="fe-section">
  <div className="fe-section-header">
    <span className="fe-section-number">4</span>
    <h2 className="fe-section-title">Every Field Has States</h2>
  </div>

  <p className="fe-text">
    A field is default, then focused, then filled, sometimes disabled, sometimes in error. These are not styling whims, they are the same modality-driven state model every control in the system follows. Focus is shown with the indigo ring token, not a colour that vanishes on a coloured surface; the press and hover of the checkbox and toggle are tonal indigo layers, gated to the devices that can produce them. Rather than restate that model here, the field inherits it.
  </p>

{/* CHAPTER 5: SMART DEFAULTS                       */}
{/* ============================================ */}

<div className="fe-section">
  <div className="fe-section-header">
    <span className="fe-section-number">5</span>
    <h2 className="fe-section-title">The Fastest Field Is the One Already Filled</h2>
  </div>

  <p className="fe-text">
    The best way to reduce the cost of a field is to remove it, and the next best is to answer it in advance. A smart default (the country pre-selected from locale, the event date defaulting to the next weekend, the quantity starting at one) turns an act of typing into an act of confirming. Defaults are powerful precisely because most people accept them, which is exactly why they carry an ethical weight: the default must be the choice the user would most likely make with full attention, and it must always stay editable.
  </p>

  <p className="fe-text">
    That line, between a helpful default and a manipulative one, is not a form-styling question, it is a persuasion question. A default that quietly opts the user into a paid add-on is the same mechanism pointed the wrong way. The design-system rule is simple: pre-fill to save effort, never to extract a choice the user did not mean to make.
  </p>

  <a className="fe-xlink" href="./?path=/docs/patterns-persuasion--docs">
    <span className="fe-material" aria-hidden="true">arrow_forward</span>
    See Patterns / Persuasion for the ethics of defaults and the truth-and-reversibility test
  </a>
</div>

    Accessible forms are not a separate build, they are the same build done correctly. Three WCAG success criteria govern most of it, and each one maps directly to a habit from the chapters above.

<div className="fe-section">
  <div className="fe-section-header">
    <span className="fe-section-number">7</span>
    <h2 className="fe-section-title">Rules of the Form</h2>
  </div>

  <div className="fe-dodont">
    <div className="fe-list do">
      <div className="fe-list-head">
        <span className="fe-material" style={{ fontSize: '1.25rem' }} aria-hidden="true">check_circle</span>
        Do
      </div>
      <ul>
        <li>Keep the label visible above every field, always.</li>
        <li>One field per line. A single column reads top to bottom.</li>
        <li>Match <code>type</code> and <code>inputmode</code> to the answer, so the right keyboard appears.</li>
        <li>Add the <code>autocomplete</code> token so the browser can fill it.</li>
        <li>Reward valid input while typing, surface errors on blur.</li>
        <li>Write errors that say how to fix the problem, in text.</li>
        <li>Use <code>--color-error</code> for errors and nothing else.</li>
        <li>Reach for the native control before building a custom one.</li>
      </ul>
    </div>
    <div className="fe-list dont">
      <div className="fe-list-head">
        <span className="fe-material" style={{ fontSize: '1.25rem' }} aria-hidden="true">cancel</span>
        Don't
      </div>
      <ul>
        <li>Use a placeholder as the label. It vanishes on the first keystroke.</li>
        <li>Put two fields side by side to look compact.</li>
        <li>Leave everything <code>type="text"</code> and force a keyboard hunt on mobile.</li>
        <li>Validate on every keystroke and flag half-typed input as wrong.</li>
        <li>Ship "Invalid" or "Error" with no way to recover.</li>
        <li>Signal an error with a red border and no words.</li>
        <li>Tint unrelated UI red, or introduce any second accent colour.</li>
        <li>Rebuild a date picker or select the platform already ships.</li>
      </ul>
    </div>
  </div>

  <p className="fe-text" style={{ marginTop: '1.5rem' }}>
    <strong>Decision ladder.</strong> Adding a field? Stop at the first rung that resolves it.
  </p>

  <div className="fe-ladder">
    <div className="fe-ladder-step">
      <p className="fe-ladder-text"><strong>Does this field need to exist?</strong> Every field is a cost. If you can infer it, default it, or drop it, do that before you style anything.</p>
    </div>
    <div className="fe-ladder-step">
      <p className="fe-ladder-text"><strong>Can a native control answer it?</strong> A <code>&lt;select&gt;</code>, <code>type="date"</code>, a checkbox. Use it, and the keyboard and accessibility come built in.</p>
    </div>
    <div className="fe-ladder-step">
      <p className="fe-ladder-text"><strong>What is the answer's shape?</strong> Pick the <code>type</code> and <code>inputmode</code> that raise the matching keyboard, and add the <code>autocomplete</code> token.</p>
    </div>
    <div className="fe-ladder-step">
      <p className="fe-ladder-text"><strong>What can go wrong, and how is it fixed?</strong> Write the help text and the recovery message before the layout. If you cannot say how to fix it, the rule is wrong, not the user.</p>
    </div>
    <div className="fe-ladder-step">
      <p className="fe-ladder-text"><strong>Only now, assemble it</strong> from the primitives above: visible label, control, help, error, in one column, states inherited from Patterns / Interactive.</p>
    </div>
  </div>
