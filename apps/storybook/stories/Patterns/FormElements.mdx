<div className="fe-section">
  <div className="fe-section-header">
    <span className="fe-section-number">1</span>
    <h2 className="fe-section-title">Native-first, and doubly so on mobile</h2>
  </div>

  <p className="fe-text">
    The default is always the plain HTML control with the right <code>type</code>, <code>inputmode</code> and <code>autocomplete</code>. On a phone this is not a preference, it is leverage: a semantic <code>&lt;input type&gt;</code> raises the correct on-screen keyboard, opens the platform picker, and arrives with keyboard support, focus handling and screen-reader semantics already built in. MDN is blunt about it: prefer the semantic input type over hacks, for better accessibility and built-in validation.
  </p>

  <p className="fe-text">
    Go custom only when one of two things is true: the pattern <strong>does not exist in HTML</strong> (a real multi-select, a filtering combobox, tags, rating), or the brand <strong>must style something the platform will not let you style</strong> (the popup of a <code>&lt;select&gt;</code>, the calendar of a date picker). <div className="fe-section">
  <div className="fe-section-header">

  <p className="fe-text">
    A field is four parts, and dropping any one shifts the work onto the user. The <strong>label</strong> asks and stays visible. The <strong>control</strong> takes the answer. <strong>Help</strong> pre-empts the mistake. The <strong>error</strong> appears only when something is wrong, and says exactly how to fix it.

  <div className="fe-anatomy">
    <div className="fe-anatomy-part">

{/* 4: VALIDATION                                 */}
{/* ============================================ */}

<div className="fe-section">
  <div className="fe-section-header">
    <span className="fe-section-number">4</span>
    <h2 className="fe-section-title">Validate at the right moment, in the right words</h2>
  </div>

  <p className="fe-text">
    Start from HTML constraint validation, <code>required</code>, <code>type</code>, <code>pattern</code>, <code>min</code>, <code>max</code>, <code>step</code>. It is free, works without JavaScript, and is the source of truth. Present its state with the newer <code>:user-invalid</code> and <code>:user-valid</code> pseudo-classes, not <code>:invalid</code> and <code>:valid</code>. The user variants only match after the person has interacted or tried to submit, which is exactly what stops a pristine form from lighting up red on load.
  </p>

  <div className="fe-code" dangerouslySetInnerHTML={{ __html:
`<span class="fe-c-com">/* matches only after interaction, never on first paint */</span>
<span class="fe-c-sel">input:user-invalid</span> { <span class="fe-c-prop">border-color</span>: var(--color-error); }
<span class="fe-c-sel">input:user-valid</span>   { <span class="fe-c-prop">border-color</span>: var(--color-success-strong); }` }} />

  <p className="fe-text">
    On timing, the rule is <strong>reward early, punish late</strong>. Confirm a field as valid while the user types, but hold a new error until they leave it. Flagging the first letter of an email as a broken email is technically true and deeply unhelpful.
  </p>

  <p className="fe-text">
    The rule has a second half that starts the moment a field has <em>already</em> shown an error. From then on, switch that field to re-validate on every keystroke and <strong>clear the error the instant the value becomes valid</strong>. Once someone is fixing a known problem, the reassurance that they succeeded should arrive immediately, never held back until they leave and re-enter the field.
  </p>

  <div className="fe-timeline">
    <div className="fe-tl-step">
      <span className="fe-tl-when"><span className="fe-material" style={{ fontSize: '0.85rem' }} aria-hidden="true">keyboard</span> While typing</span>
      <div className="fe-tl-title">Reward, never punish</div>
      <div className="fe-tl-body">Show a success tick the moment input becomes valid. Hold the first error back; an incomplete field is not yet a wrong one. After a field has erred, though, re-check on input and clear it the instant it is valid.</div>
    </div>
    <div className="fe-tl-step">
      <span className="fe-tl-when"><span className="fe-material" style={{ fontSize: '0.85rem' }} aria-hidden="true">logout</span> On blur</span>
      <div className="fe-tl-title">Now you may correct</div>
      <div className="fe-tl-body">When focus leaves the field, the answer is final enough to judge. Surface the inline error, right under the field.</div>
    </div>
    <div className="fe-tl-step">
      <span className="fe-tl-when"><span className="fe-material" style={{ fontSize: '0.85rem' }} aria-hidden="true">send</span> On submit</span>
      <div className="fe-tl-title">Catch the rest</div>
      <div className="fe-tl-body">Re-check everything, move focus to the first error, summarise in a live region. Never let a form fail silently.</div>
    </div>
  </div>

  <p className="fe-text">
    A good error message is <strong>visible and specific</strong>, says <strong>what went wrong in plain language</strong>, and tells the user <strong>how to fix it</strong>. "Invalid input" fails all three. "Enter a date after today, the event cannot start in the past" passes all three. WCAG 3.3.1 requires the error to be identified in text, so the field carries <code>aria-invalid="true"</code> and points at the message with <code>aria-describedby</code>. The colour reinforces; the words are the message.
  </p>

  <div className="fe-lab">
    <div className="fe-demo">
      <div className="fe-demo-label">Reward: valid while typing</div>
      <div className="fe-field">
        <label className="fe-label" htmlFor="fe-ok">Email address</label>
        <input id="fe-ok" className="fe-input success" type="email" defaultValue="ana@ramoslabs.co" aria-describedby="fe-ok-msg" />
        <span id="fe-ok-msg" className="fe-success-text"><span className="fe-material" aria-hidden="true">check_circle</span> Looks good.</span>
      </div>
    </div>
    <div className="fe-demo">
      <div className="fe-demo-label">Punish late: actionable error on blur</div>
      <div className="fe-field">
        <label className="fe-label" htmlFor="fe-bad">Email address</label>
        <input id="fe-bad" className="fe-input error" type="email" defaultValue="ana@ramoslabs" aria-invalid="true" aria-describedby="fe-bad-msg" />
        <span id="fe-bad-msg" className="fe-error"><span className="fe-material" aria-hidden="true">error</span> Add the domain ending, like .co or .com, so the address can receive mail.</span>
      </div>
    </div>
  </div>

  <div className="fe-callout">
    <span className="fe-material" aria-hidden="true">palette</span>
    <p className="fe-callout-text">
      <strong>Errors use <code>--color-error</code>, the only warm colour on the page.</strong> Everything interactive stays indigo; red's single job is to mark a genuine problem. Do not tint help text, borders or icons red for emphasis. Reserve it, so its appearance always means something is wrong.
    </p>
  </div>
</div>

<div className="fe-section">

  <p className="fe-text">

<div className="fe-section" style={{ borderBottom: 'none' }}>
  <div className="fe-rules" style={{ marginTop: '1.5rem' }}>
    <div className="fe-rule">
      <p className="fe-rule-line"><span className="fe-tag rule">Rule</span> Reach for the native control before building a custom one.</p>
      <p className="fe-rule-why"><span className="fe-tag why">Why</span> A native <code>&lt;select&gt;</code>, date input or checkbox ships the keyboard, picker and accessibility a hand-built widget has to re-earn from scratch.</p>
    </div>
    <div className="fe-rule">
      <p className="fe-rule-line"><span className="fe-tag rule">Rule</span> Brand native controls with <code>accent-color</code>, <code>caret-color</code> and <code>color-scheme</code>, and know their reach.</p>
      <p className="fe-rule-why"><span className="fe-tag why">Why</span> They tint checkbox, radio, range, progress and the caret without a rebuild; select, date and text they do not touch, so plan for those.</p>
    </div>
    <div className="fe-rule">
      <p className="fe-rule-line"><span className="fe-tag rule">Rule</span> Keep the label visible above every field, and mark the rarer of required or optional in words.</p>
      <p className="fe-rule-why"><span className="fe-tag why">Why</span> A placeholder vanishes on the first keystroke and takes the question with it; an asterisk alone reaches no screen reader.</p>
    </div>
    <div className="fe-rule">
      <p className="fe-rule-line"><span className="fe-tag rule">Rule</span> Match <code>type</code>, <code>inputmode</code>, <code>enterkeyhint</code> and <code>autocomplete</code> to the answer.</p>
      <p className="fe-rule-why"><span className="fe-tag why">Why</span> The right keyboard rises on the first tap and a saved profile fills the field in one, instead of a manual retype.</p>
    </div>
    <div className="fe-rule">
      <p className="fe-rule-line"><span className="fe-tag rule">Rule</span> Render inputs at 16px or larger and give interactive targets at least 24px, ideally 44px.</p>
      <p className="fe-rule-why"><span className="fe-tag why">Why</span> Below 16px iOS Safari auto-zooms on focus; below 24px the target fails WCAG 2.5.8, and the thumb misses.</p>
    </div>
    <div className="fe-rule">
      <p className="fe-rule-line"><span className="fe-tag rule">Rule</span> Reward valid input while typing with <code>:user-valid</code>, and surface errors on blur.</p>
      <p className="fe-rule-why"><span className="fe-tag why">Why</span> Flagging a half-typed value as wrong punishes progress; once a field has already erred, re-check on input and clear it the instant it is valid.</p>
    </div>
    <div className="fe-rule">
      <p className="fe-rule-line"><span className="fe-tag rule">Rule</span> Write errors that name the fix, in text, with <code>aria-invalid</code> and <code>aria-describedby</code>.</p>
      <p className="fe-rule-why"><span className="fe-tag why">Why</span> "Invalid" leaves no way to recover, and a red border alone reaches no screen reader; the words carry the message.</p>
    </div>
    <div className="fe-rule">
      <p className="fe-rule-line"><span className="fe-tag rule">Rule</span> Reserve <code>--color-error</code> for genuine errors and nothing else.</p>
      <p className="fe-rule-why"><span className="fe-tag why">Why</span> One warm colour on an otherwise indigo page means its appearance always signals a real problem.</p>
    </div>
    <div className="fe-rule">
      <p className="fe-rule-line"><span className="fe-tag rule">Rule</span> When you must go custom, build against the matching WAI-ARIA APG pattern and keep a native base underneath.</p>
      <p className="fe-rule-why"><span className="fe-tag why">Why</span> The APG pattern encodes the keyboard and roles users expect, and the native base keeps the field working if the script fails.</p>
    </div>
  </div>

  <p className="fe-text" style={{ marginTop: '1.5rem' }}>
    <strong>Decision ladder.</strong> Adding a field? Stop at the first rung that resolves it.
  </p>

  <div className="fe-ladder">
    <div className="fe-ladder-step">
      <p className="fe-ladder-text"><strong>Does this field need to exist?</strong> Every field is a cost. If you can infer it, default it, or drop it, do that before styling anything.</p>
    </div>
    <div className="fe-ladder-step">
      <p className="fe-ladder-text"><strong>Can a native control answer it?</strong> A <code>&lt;select&gt;</code>, <code>type="date"</code>, a checkbox. Use it, and keyboard and accessibility come built in.</p>
    </div>
    <div className="fe-ladder-step">
      <p className="fe-ladder-text"><strong>What is the answer's shape?</strong> Pick the <code>type</code> and <code>inputmode</code> that raise the matching keyboard, add <code>enterkeyhint</code> and the <code>autocomplete</code> token, and brand it with <code>accent-color</code>.</p>
    </div>
    <div className="fe-ladder-step">
      <p className="fe-ladder-text"><strong>What can go wrong, and how is it fixed?</strong> Write the help and the recovery message before the layout. If you cannot say how to fix it, the rule is wrong, not the user.</p>
    </div>
    <div className="fe-ladder-step">
      <p className="fe-ladder-text"><strong>Only now, assemble it</strong> from the primitives: visible label, control, help, error, in one column, states inherited from Patterns / Interactive. If the pattern is not in HTML, build the matching APG pattern over a native base.</p>
    </div>
  </div>
</div>
