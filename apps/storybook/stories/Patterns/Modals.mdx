  </div>

  <p className="md-text">
    An accessible modal dialog is not a styled overlay. It is a contract with the keyboard and the screen reader, and the WAI-ARIA Authoring Practices name six terms of it. Miss any one and the modal is broken for someone. The good news is in Section 2: the native element hands you all six.
  </p>

  <div className="md-reqs">

    <div className="md-req">
      <div className="md-req-head">
        <span className="md-req-num">1</span>
        <span className="md-req-title">It has an accessible name</span>
      </div>
      <p className="md-req-body">Point <code>aria-labelledby</code> at the visible title, or set <code>aria-label</code> when there is no visible title. Without a name it fails WCAG 4.1.2.</p>
      <p className="md-req-cite"><b>APG, Dialog Pattern.</b> The dialog has "a value set for the aria-labelledby property that refers to a visible dialog title."</p>
    </div>

    <div className="md-req">
      <div className="md-req-head">
        <span className="md-req-num">2</span>
        <span className="md-req-title">Focus moves in on open</span>
      </div>
      <p className="md-req-body">When the dialog opens, focus lands on an element inside it, not out on the page behind. Use <code>autofocus</code> to place it deliberately.</p>
      <p className="md-req-cite"><b>APG, Dialog Pattern.</b> "When a dialog opens, focus moves to an element inside the dialog."</p>
    </div>

    <div className="md-req">
      <div className="md-req-head">
        <span className="md-req-num">3</span>
        <span className="md-req-title">Tab cycles inside</span>
      </div>
      <p className="md-req-body">Tab off the last control wraps to the first, and Shift+Tab off the first wraps to the last. Focus never escapes to the inert page.</p>
      <p className="md-req-cite"><b>APG, Dialog Pattern.</b> On the last tabbable element, Tab "moves focus to the first tabbable element inside the dialog."</p>
    </div>

    <div className="md-req">
      <div className="md-req-head">
        <span className="md-req-num">4</span>
        <span className="md-req-title">Escape closes it</span>
      </div>
      <p className="md-req-body">The Escape key dismisses the dialog. This is the release valve that keeps the focus trap legal under WCAG 2.1.2 (see Section 6).</p>
      <p className="md-req-cite"><b>APG, Dialog Pattern.</b> "Escape: Closes the dialog."</p>
    </div>

    <div className="md-req">
      <div className="md-req-head">
        <span className="md-req-num">5</span>
        <span className="md-req-title">Focus returns to the trigger</span>
      </div>
      <p className="md-req-body">On close, focus goes back to the control that opened the dialog, so the user resumes where they left off rather than at the top of the page.</p>
      <p className="md-req-cite"><b>APG, Dialog Pattern.</b> "Focus returns to the element that invoked the dialog."</p>
    </div>

    <div className="md-req">
      <div className="md-req-head">
        <span className="md-req-num">6</span>
        <span className="md-req-title">The background is inert</span>
      </div>
      <p className="md-req-body">Everything behind the dialog leaves the tab order and the accessibility tree. A pointer, a Tab, and a screen reader all stop at the dialog edge.</p>
      <p className="md-req-cite"><b>APG, Dialog Pattern.</b> "Windows under a modal dialog are inert... users cannot interact with content outside an active dialog window."</p>
    </div>

  </div>
</div>

{/* 3 - DIALOG VS ALERTDIALOG                     */}

<div className="md-section">

{/* 4 - WHEN NOT TO USE A MODAL                   */}
{/* ============================================ */}

<div className="md-section">
  <div className="md-section-header">
    <span className="md-section-number">4</span>
    <h2 className="md-section-title">When not to use a modal at all</h2>
  </div>

  <p className="md-text">
    A modal charges a tax. NN/g lists the bill: it demands immediate action, breaks the flow, makes the user lose their place, adds cognitive load to recover the task, and hides relevant background. "Modal dialogs that are not directly related to users' goals are perceived as annoying and can diminish trust." Before you open one, walk down this ladder and stop at the first rung that fits. Most of the time you stop before the modal.
  </p>

  <div className="md-ladder">
    <div className="md-rung">
      <span className="md-rung-step">1</span>
      <p className="md-rung-body"><b>A separate page.</b> For anything complex or multi-step. This is GOV.UK's default: their "one thing per page" philosophy sends the user to another page instead of a modal, and their public design system ships no modal component at all. Simplicity over speed.</p>
    </div>
    <div className="md-rung">
      <span className="md-rung-step">2</span>
      <p className="md-rung-body"><b>Inline content.</b> A field failed validation? Show the error inline, next to the field. Help text, hints, and single-field errors belong in the page, never behind an interruption.</p>
    </div>
    <div className="md-rung">
      <span className="md-rung-step">3</span>
      <p className="md-rung-body"><b>A non-modal dialog.</b> When the user should keep working with the background, like a compose window, open a dialog with <code>show()</code> instead of <code>showModal()</code>. No trap, no inert, the page stays live around it.</p>
    </div>
    <div className="md-rung">
      <span className="md-rung-step">4</span>
      <p className="md-rung-body"><b>A popover.</b> Action menus, toasts, form suggestions, teaching hints. Lightweight, non-modal, light-dismiss. This is Section 5.</p>
    </div>
  </div>

  <div className="md-callout warn">
    <span className="md-material" aria-hidden="true">block</span>
    <p className="md-callout-text">
      <strong>Never modal.</strong> Newsletter and promo interrupts unrelated to the goal, high-risk checkout (NN/g advises against it explicitly), decisions that need information the user cannot see inside the modal, and single-field validation errors. If the content does not deserve to steal the room, it does not get a modal.
    </p>
  </div>
</div>

{/* 6 - PITFALLS                                  */}
{/* ============================================ */}

<div className="md-section">
  <div className="md-section-header">
    <span className="md-section-number">6</span>
    <h2 className="md-section-title">The pitfalls, and the WCAG nuance</h2>
  </div>

  <p className="md-text">
    Every one of these is a real dialog shipped broken. The native element prevents most of them; a hand-built modal has to guard against all of them.
  </p>

  <div className="md-rules">
    <div className="md-rule">
      <p className="md-rule-line"><span className="md-rule-tag rule">Trap</span>Focus escapes to the background. Tab lands on links behind the dialog. The native element inertizes the rest of the document for you; by hand you must set the <code>inert</code> attribute, not just <code>pointer-events</code>.</p>
    </div>
    <div className="md-rule">
      <p className="md-rule-line"><span className="md-rule-tag rule">Trap</span>Focus never returns to the trigger, most often because the open or close was animated, which drops the automatic return. Restore it in an event listener.</p>
    </div>
    <div className="md-rule">
      <p className="md-rule-line"><span className="md-rule-tag rule">Trap</span>No accessible name. Missing <code>aria-labelledby</code> or <code>aria-label</code> fails WCAG 4.1.2 outright.</p>
    </div>
    <div className="md-rule">
      <p className="md-rule-line"><span className="md-rule-tag rule">Trap</span>Close only on outside click, with no keyboard exit. Always ship an explicit close button plus Escape.</p>
    </div>
    <div className="md-rule">
      <p className="md-rule-line"><span className="md-rule-tag rule">Trap</span>Opened with the <code>open</code> attribute, which produces a non-modal, no trap and no Escape and no inert. Use <code>.showModal()</code>.</p>
    </div>
    <div className="md-rule">
      <p className="md-rule-line"><span className="md-rule-tag rule">Trap</span>Nesting modals. Stacked dialogs confuse which Escape closes what. Prefer one dialog, or steps inside a single dialog.</p>
    </div>
  </div>

  <div className="md-callout">
    <span className="md-material" aria-hidden="true">verified</span>
    <p className="md-callout-text">
      <strong>The trap is legal, on one condition.</strong> A modal's focus trap looks like it violates WCAG 2.1.2 No Keyboard Trap, and it does not, as long as there is a way out. W3C: restricting focus to a subsection "does not fail the requirements of this criterion, as long as the user knows how to 'untrap' the focus and leave that component." Trap the focus on purpose, but Escape or a Cancel button must always release it. A modal with no keyboard exit does fail 2.1.2.
    </p>
  </div>
</div>

{/* 7 - RESPONSIVE BRIDGE                         */}
{/* ============================================ */}

<div className="md-section">
  <div className="md-section-header">
    <span className="md-section-number">7</span>
    <h2 className="md-section-title">On a phone, the modal becomes a sheet</h2>
  </div>

  <p className="md-text">
    A centered modal is a desktop shape. On a phone it usually becomes a bottom sheet, a panel that rises from the bottom edge into the thumb's reach. What changes is only the presentation, position, anchor, and entry animation. The semantics do not move. A bottom sheet is still a modal <code>dialog</code> or <code>alertdialog</code>, and it owes every one of the six requirements from Section 1: an accessible name, focus in, a trapped Tab cycle, Escape to close, focus back to the trigger, and an inert background. The same native <code>&lt;dialog&gt;</code> can present as a sheet through CSS while keeping <code>showModal()</code> and its guarantees.
  </p>

  <div className="md-callout">
    <span className="md-material" aria-hidden="true">swipe_up</span>
    <p className="md-callout-text">
      <strong>Do not trade a11y for the gesture.</strong> Drag-to-dismiss is a pointer nicety, not a substitute for the keyboard. Every sheet still needs Escape and a visible close button, and the drag must have a keyboard equivalent. The presentation is documented in <strong>Patterns / Mobile First</strong>; the doctrine on this page travels with it unchanged.
    </p>
  </div>
</div>

{/* ============================================ */}
{/* 8 - THE RULES                                 */}
{/* ============================================ */}

<div className="md-section">
  <div className="md-section-header">
    <span className="md-section-number">8</span>
    <h2 className="md-section-title">The rules</h2>
  </div>

  <p className="md-text">
    Each rule is stated in the positive, with the reason it exists. Where the wrong turn is easy to make, a recommended and an avoid card sit side by side.
  </p>

  <div className="md-rules">

    <div className="md-rule">
      <p className="md-rule-line"><span className="md-rule-tag rule">Rule</span>Reach for the native <code>&lt;dialog&gt;</code> with <code>.showModal()</code> before building anything by hand.</p>
      <p className="md-rule-line"><span className="md-rule-tag why">Why</span>It delivers all six accessibility requirements correctly for free: focus in, Tab trapped, Escape, inert background, focus return, top layer. A hand-built modal has to re-derive each one, which is where the bugs live.</p>
      <div className="md-ra">
        <div className="md-ra-card reco">
          <div className="md-ra-head"><span className="md-material" aria-hidden="true">check_circle</span>Recommended</div>
          <p className="md-ra-body">Open with <code>.showModal()</code>, close a form with <code>method="dialog"</code>, style the dim with <code>::backdrop</code>.</p>
        </div>
        <div className="md-ra-card avoid">
          <div className="md-ra-head"><span className="md-material" aria-hidden="true">block</span>Avoid</div>
          <p className="md-ra-body">Opening with the <code>open</code> attribute, which yields a non-modal with no trap, no Escape, and no inert.</p>
        </div>
      </div>
    </div>

    <div className="md-rule">
      <p className="md-rule-line"><span className="md-rule-tag rule">Rule</span>Give every modal an accessible name and a real close button, and never put <code>tabindex</code> on the <code>&lt;dialog&gt;</code>.</p>
      <p className="md-rule-line"><span className="md-rule-tag why">Why</span>An unnamed dialog fails WCAG 4.1.2, and Escape alone is not discoverable. MDN calls a visible close button the most robust exit for all users, and warns the dialog itself is not interactive.</p>
    </div>

    <div className="md-rule">
      <p className="md-rule-line"><span className="md-rule-tag rule">Rule</span>Use <code>role="alertdialog"</code> for destructive or irreversible actions, with <code>aria-describedby</code> and initial focus on Cancel.</p>
      <p className="md-rule-line"><span className="md-rule-tag why">Why</span>The role lets assistive tech treat the interruption specially, and focusing Cancel makes the dangerous choice deliberate rather than one stray Enter away.</p>
      <div className="md-ra">
        <div className="md-ra-card reco">
          <div className="md-ra-head"><span className="md-material" aria-hidden="true">check_circle</span>Recommended</div>
          <p className="md-ra-body"><code>alertdialog</code> with the message wired to <code>aria-describedby</code> and <code>autofocus</code> on Cancel.</p>
        </div>
        <div className="md-ra-card avoid">
          <div className="md-ra-head"><span className="md-material" aria-hidden="true">block</span>Avoid</div>
          <p className="md-ra-body">A plain <code>dialog</code> for a delete, or <code>autofocus</code> on the destructive button so one Enter wipes the data.</p>
        </div>
      </div>
    </div>

    <div className="md-rule">
      <p className="md-rule-line"><span className="md-rule-tag rule">Rule</span>Before opening a modal, try a page, an inline message, a non-modal dialog, or a popover first.</p>
      <p className="md-rule-line"><span className="md-rule-tag why">Why</span>A modal steals context and demands an immediate answer. Most content does not earn that cost, and an interruption unrelated to the goal reads as annoying and erodes trust.</p>
    </div>

    <div className="md-rule">
      <p className="md-rule-line"><span className="md-rule-tag rule">Rule</span>Keep the focus trap, but always leave a keyboard exit, and if you animate, restore the focus return by hand.</p>
      <p className="md-rule-line"><span className="md-rule-tag why">Why</span>The trap is legal under WCAG 2.1.2 only while the user can untrap with Escape or Cancel, and animating a dialog drops the automatic focus return that the six requirements depend on.</p>
    </div>

    <div className="md-rule">
      <p className="md-rule-line"><span className="md-rule-tag rule">Rule</span>Reserve the <code>--color-error</code> family for the destructive path and keep everything else mono-indigo; radius 6px on buttons, 12px on the dialog.</p>
      <p className="md-rule-line"><span className="md-rule-tag why">Why</span>Red carries a single meaning, danger, so it stays out of chrome and buttons that do not destroy. The destructive fill uses red 600 because red 500 clears only 3.76:1 under white text.</p>
    </div>

  </div>
</div>
